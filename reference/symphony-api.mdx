The Symphony HTTP server starts automatically when you run `symphony WORKFLOW.md`. By default it binds to `http://127.0.0.1:8080`. Override the port with `--port` or `server.port` in your WORKFLOW.md. Set `server.host: 0.0.0.0` to bind all interfaces.

**Base URL:** `http://localhost:<port>` (default `8080`)

---

## `GET /`

HTML dashboard. Auto-refreshes every 2 seconds.

Shows: running sessions with turn count, token usage, and last activity; retry queue; shared context table (author, scope, preview, age, TTL); supervisor status and counters; completed issues; polling stats; rate limits; and a link to the Linear project.

## `GET /api/v1/events`

WebSocket live event stream of `SymphonyEventEnvelope` values. Connect with any WebSocket client.

```bash
websocat "ws://127.0.0.1:8080/api/v1/events?issue=KAT-920&type=worker,tool&severity=info"
```

### Query parameters

<ParamField query="issue" type="string">
  Comma-separated issue identifiers to filter events to. Example: `KAT-920,KAT-921`.
</ParamField>

<ParamField query="type" type="string">
  Comma-separated event type names to include. Example: `worker,tool,escalation_created`.
</ParamField>

<ParamField query="severity" type="string">
  Comma-separated severity levels to include. Example: `warn,error`.
</ParamField>

### Filter semantics

Filters are **OR within each field** and **AND across fields**:

- `issue=KAT-920,KAT-921` matches either issue
- `type=worker,tool` matches either event kind
- `severity=warn,error` matches either severity level
- When multiple fields are provided, all field filters must match

### Connection events

| Event | Description |
|-------|-------------|
| `ws_client_connected` | New WebSocket client connected |
| `ws_client_disconnected` | Client disconnected; `reason` field gives the cause |
| `ws_event_dropped` | Event dropped due to backpressure |
| `ws_heartbeat_sent` | Periodic heartbeat emitted |

Disconnect reasons: `backpressure`, `server_shutdown`, `client_closed`, `protocol_error`.

## `POST /api/v1/escalations/:request_id/respond`

Resolve a pending escalation. The waiting worker resumes its session with the provided response â€” no restart required.

### Path parameters

<ParamField path="request_id" type="string" required>
  The `request_id` from the escalation record.
</ParamField>

### Request body

```json
{
  "response": "Use the PostgreSQL approach",
  "responder_id": "alice"
}
```

<ParamField body="response" type="any" required>
  The response to forward to the waiting worker.
</ParamField>

<ParamField body="responder_id" type="string">
  Optional identifier of the person responding.
</ParamField>

### Response codes

| Code | Meaning |
|------|---------|
| `200` | Escalation resolved successfully |
| `404` | Escalation not found or already expired |
| `409` | Escalation already resolved |

### Escalation lifecycle

1. Worker emits an interactive `extension_ui_request`
2. Symphony emits `escalation_created` on `/api/v1/events`
3. Escalation appears in `/api/v1/escalations` and the dashboard/TUI
4. Client calls `POST /api/v1/escalations/:request_id/respond`
5. Symphony emits `escalation_responded`, `escalation_timed_out`, or `escalation_cancelled`
6. Response is forwarded to the waiting worker

## `GET /api/v1/:issue_identifier`

Per-issue running or retry projection.

`:issue_identifier` must match the pattern `TEAM-NNN` (uppercase prefix, hyphen, digits). Returns `404` when the issue is not currently running or in the retry queue.

### Response

```json
{
  "issue": {
    "issue_id": "issue-id-123",
    "issue_identifier": "ENG-42",
    "status": "running",
    "attempt": 1,
    "error": null,
    "worker_host": "worker1.example.com",
    "workspace_path": "/tmp/symphony_workspaces/ENG-42"
  }
}
```

## `POST /api/v1/refresh`

Request an immediate poll. Multiple concurrent requests are coalesced into a single poll.

Returns `202 Accepted`.

### Response

```json
{
  "queued": true,
  "coalesced": false,
  "pending_requests": 1
}
```

<ResponseField name="queued" type="boolean">Whether the poll was queued.</ResponseField>
<ResponseField name="coalesced" type="boolean">Whether this request was merged with an already-pending poll.</ResponseField>
<ResponseField name="pending_requests" type="number">Number of requests that were coalesced into this poll.</ResponseField>

## `POST /api/v1/context`

Write a shared context entry. Workers automatically receive the most relevant recent entries in their prompt preamble.

### Request body

```json
{
  "author_issue": "KAT-42",
  "scope": "project",
  "content": "Use the v2 API endpoint for all new integrations.",
  "ttl_ms": 86400000
}
```

<ParamField body="author_issue" type="string" required>
  Identifier of the issue writing this entry.
</ParamField>

<ParamField body="scope" type="string" required>
  Scope for this entry. Used to match relevant entries to workers.
</ParamField>

<ParamField body="content" type="string" required>
  The context content.
</ParamField>

<ParamField body="ttl_ms" type="number">
  Time-to-live in milliseconds. Defaults to `shared_context.ttl_ms` from WORKFLOW.md (default: 24 hours).
</ParamField>

---

## `DELETE /api/v1/context/{id}`

Delete a specific shared context entry by ID.

---

## `DELETE /api/v1/context`

Clear shared context entries. When a `scope` query parameter is provided, only entries matching that scope are deleted. When omitted, all entries are cleared.

### Query parameters

<ParamField query="scope" type="string">
  Scope to clear. Omit to clear all entries.
</ParamField>
